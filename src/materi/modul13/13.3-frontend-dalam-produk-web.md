## Apa itu frontend?
Frontend adalah bagian aplikasi yang berinteraksi langsung dengan pengguna di browser. Semua yang terlihat, diklik, dibaca, diisi, dan dirasakan pengguna berada di wilayah frontend. HTML membentuk struktur, CSS mengatur tampilan, dan JavaScript menambahkan interaksi. Framework seperti React, Vue, Svelte, atau Angular membantu membangun aplikasi yang lebih kompleks dengan pola komponen dan state management.

Namun frontend bukan hanya membuat tampilan sesuai desain. Frontend bertanggung jawab memastikan halaman cepat dimuat, aksesibel, responsif di berbagai ukuran layar, nyaman digunakan, dan mampu berkomunikasi dengan backend. Frontend adalah titik temu antara UI/UX, data, dan teknologi browser.

## Tanggung jawab utama frontend developer
Tanggung jawab pertama adalah menerjemahkan desain menjadi antarmuka yang akurat dan fleksibel. Akurat berarti sesuai dengan kebutuhan visual dan interaksi. Fleksibel berarti tetap rapi ketika konten berubah, teks lebih panjang, data kosong, atau perangkat berbeda. Developer tidak boleh hanya membuat halaman terlihat bagus pada satu ukuran layar dan satu data contoh.

Tanggung jawab kedua adalah mengelola interaksi. Dropdown, modal, validasi form, navigasi, pencarian, filter, tabs, drag-and-drop, dan animasi membutuhkan logika yang jelas. Interaksi harus memberi feedback dan tidak membingungkan. Jika sebuah tombol memicu proses, tampilkan loading. Jika input salah, tampilkan pesan. Jika aksi berhasil, beri konfirmasi.

Tanggung jawab ketiga adalah integrasi data. Frontend mengambil data dari API, menampilkan status loading, menangani error, mengirim form, dan menjaga agar tampilan sesuai kondisi terbaru. Frontend juga perlu memahami batas keamanan: jangan menyimpan secret di browser, jangan mempercayai input hanya karena sudah divalidasi di client, dan jangan menampilkan data sensitif sembarangan.

## Komponen sebagai cara berpikir
Aplikasi frontend modern biasanya dibangun dari komponen. Komponen adalah bagian UI yang punya tanggung jawab tertentu dan dapat digabungkan menjadi halaman. Contohnya Button, Card, Navbar, CourseList, ProgressBar, dan UserMenu. Komponen yang baik mudah dipahami, menerima props yang jelas, dan tidak melakukan terlalu banyak hal sekaligus.

Cara berpikir komponen membantu mengurangi duplikasi. Jika tombol dipakai di banyak tempat, buat satu komponen Button dengan varian yang konsisten. Jika card materi muncul di dashboard dan halaman modul, gunakan komponen yang sama atau pola yang mirip. Ini membuat aplikasi lebih mudah dirawat dan tampilan lebih konsisten.

## State dan data di frontend
State adalah data yang berubah dan memengaruhi tampilan. Contohnya nilai input, status modal, halaman aktif, data user, atau progres belajar. State lokal cukup untuk kebutuhan kecil. Context atau global store berguna ketika data dipakai banyak bagian. Server-state library membantu ketika data dari API perlu caching, refetch, dan sinkronisasi.

Frontend developer perlu tahu kapan state harus disimpan, kapan cukup dihitung dari data lain, dan kapan harus dikirim ke backend. Tidak semua nilai harus disimpan. Misalnya persentase progres bisa dihitung dari jumlah materi selesai dan total materi. Menyimpan data turunan tanpa alasan bisa menyebabkan inkonsistensi.

## Responsif dan aksesibel
Responsif berarti tampilan menyesuaikan berbagai ukuran layar. Ini bukan sekadar mengecilkan elemen. Kadang urutan konten perlu berubah, navigasi perlu disederhanakan, dan area klik perlu diperbesar. Mobile-first sering membantu karena kita mulai dari ruang paling terbatas, lalu menambahkan kompleksitas untuk layar lebih besar.

Aksesibilitas berarti aplikasi dapat digunakan oleh lebih banyak orang, termasuk pengguna keyboard, screen reader, atau pengguna dengan keterbatasan penglihatan. Gunakan HTML semantik, label form, alt text, kontras warna yang cukup, dan focus state yang terlihat. Aksesibilitas bukan fitur tambahan; ia bagian dari kualitas frontend.

## Performa frontend
Performa frontend dipengaruhi ukuran JavaScript, gambar, font, rendering, dan request jaringan. Halaman yang lambat membuat pengguna cepat pergi. Developer perlu memperhatikan lazy loading, code splitting, optimasi gambar, caching, dan menghindari render yang tidak perlu. Performa juga berkaitan dengan UX: aplikasi yang cepat terasa lebih profesional dan dapat dipercaya.

Gunakan Lighthouse atau performance panel untuk mengukur, tetapi jangan hanya mengejar skor. Pahami penyebabnya. Jika Largest Contentful Paint lambat, mungkin gambar hero terlalu besar. Jika interaksi lambat, mungkin JavaScript terlalu berat. Jika layout bergeser, mungkin gambar tidak punya ukuran tetap.

## Kolaborasi frontend dengan designer dan backend
Dengan designer, frontend perlu membahas komponen, state visual, responsive behavior, dan edge case. Dengan backend, frontend perlu membahas bentuk data, endpoint, error format, autentikasi, pagination, dan validasi. Komunikasi ini penting agar tidak terjadi asumsi berbeda. Misalnya frontend mengharapkan `title`, backend mengirim `name`; atau frontend butuh error per field, backend hanya mengirim pesan umum.

Dokumentasi kontrak API dan komponen UI membantu semua pihak. Bahkan catatan sederhana tentang payload, status code, dan contoh respons sudah sangat berguna. Semakin jelas kontrak, semakin sedikit bug integrasi.

## Latihan praktis
Ambil satu halaman yang sudah ada atau rencanakan halaman baru. Pecah halaman tersebut menjadi daftar komponen. Tentukan komponen mana yang reusable dan mana yang spesifik. Lalu tulis state yang dibutuhkan: local state, server state, dan derived state. Terakhir, daftar integrasi API yang diperlukan. Latihan ini membantu Anda melihat frontend sebagai sistem, bukan sekadar HTML dan CSS.
