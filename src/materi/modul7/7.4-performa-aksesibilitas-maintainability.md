## Website bagus harus terasa ringan
Performa web bukan hanya angka teknis. Performa menentukan kesan pertama pengguna. Jika halaman lambat terbuka, tombol terasa telat merespons, atau gambar membuat layout meloncat, pengguna bisa pergi sebelum memahami nilai produk Anda. Di banyak kasus, mempercepat website juga membantu SEO, konversi, dan kredibilitas brand. Karena itu, developer modern perlu memahami performa sejak awal, bukan menunggu aplikasi selesai.

Mulailah dari hal yang paling sering menyebabkan lambat: ukuran JavaScript, gambar terlalu besar, font berlebihan, request API yang tidak efisien, dan render komponen yang tidak perlu. Untuk website konten, optimasi gambar sering memberi dampak besar. Gunakan format modern, ukuran yang sesuai container, lazy loading untuk gambar di bawah layar, dan teks alternatif yang jelas. Untuk aplikasi interaktif, perhatikan bundle JavaScript. Jangan memasukkan library besar hanya untuk fungsi kecil yang bisa ditulis sendiri atau diimpor secara spesifik.

## Core Web Vitals secara sederhana
Core Web Vitals adalah metrik pengalaman pengguna yang sering dipakai untuk mengevaluasi kualitas halaman. Largest Contentful Paint mengukur seberapa cepat konten utama terlihat. Interaction to Next Paint mengukur respons halaman terhadap interaksi pengguna. Cumulative Layout Shift mengukur seberapa banyak layout bergeser secara tiba-tiba. Anda tidak harus menghafal semua detail teknis, tetapi perlu memahami penyebab umum masalahnya.

Jika konten utama lambat muncul, cek ukuran gambar hero, server response, dan render-blocking asset. Jika interaksi lambat, cek JavaScript berat, event handler yang melakukan pekerjaan terlalu banyak, atau re-render komponen besar. Jika layout sering bergeser, tetapkan ukuran gambar dan ruang iklan, hindari menyisipkan elemen di atas konten tanpa placeholder, dan pastikan font tidak menyebabkan perubahan besar saat selesai dimuat.

## Aksesibilitas adalah kualitas, bukan fitur tambahan
Aksesibilitas berarti website dapat digunakan oleh lebih banyak orang, termasuk pengguna screen reader, pengguna keyboard, orang dengan gangguan penglihatan, pengguna koneksi lambat, atau pengguna perangkat lama. Membuat website aksesibel bukan hanya kewajiban moral; ini juga meningkatkan kualitas untuk semua orang. Label form yang jelas membantu screen reader sekaligus membantu pengguna biasa. Kontras warna yang baik membuat teks nyaman dibaca di luar ruangan. Struktur heading yang rapi membuat halaman mudah dipindai.

Prinsip dasar aksesibilitas dimulai dari HTML semantik. Gunakan `button` untuk aksi, `a` untuk navigasi, `label` untuk input, dan heading berurutan untuk struktur konten. Jangan membuat semua elemen menjadi `div` yang diberi event click. Pastikan semua fitur penting bisa diakses dengan keyboard. Coba tekan Tab dari awal halaman dan lihat apakah fokus bergerak secara masuk akal. Tambahkan focus style yang terlihat agar pengguna tahu posisi mereka.

## Form yang ramah pengguna
Form adalah bagian yang sering menentukan keberhasilan aplikasi. Form yang buruk membuat pengguna gagal mendaftar, gagal membeli, atau gagal mengirim data. Gunakan label yang eksplisit, placeholder sebagai bantuan tambahan, bukan pengganti label. Tampilkan pesan error dekat field yang bermasalah, jelaskan cara memperbaikinya, dan jangan menghapus input pengguna saat validasi gagal. Untuk field wajib, beri tanda yang jelas. Untuk input sensitif seperti password, berikan opsi tampilkan/sembunyikan dan aturan yang mudah dipahami.

Validasi sebaiknya dilakukan di dua tempat: client dan server. Validasi client memberi feedback cepat, sedangkan validasi server menjaga keamanan dan integritas data. Jangan percaya input dari browser sepenuhnya karena pengguna bisa memodifikasi request. Pada sisi UI, bedakan error validasi, error koneksi, dan error server agar pesan lebih tepat.

## Maintainability: kode yang bisa dirawat
Maintainability adalah kemampuan kode untuk dipahami, diubah, dan dikembangkan tanpa menimbulkan kerusakan besar. Kode yang mudah dirawat biasanya memiliki struktur folder jelas, nama variabel deskriptif, komponen tidak terlalu besar, dan pemisahan tanggung jawab yang sehat. Jika satu komponen mengurus fetch data, validasi, layout, modal, filter, dan penyimpanan lokal sekaligus, komponen itu akan sulit diuji dan sulit diubah.

Pecah kode berdasarkan alasan perubahan. Komponen presentasi fokus pada tampilan. Hook dapat menyimpan logika state atau efek samping. File service dapat mengurus komunikasi API. Tipe data dapat diletakkan di folder types. Namun jangan terlalu cepat membuat abstraksi. Abstraksi yang baik lahir dari pola yang berulang, bukan dari tebakan bahwa sesuatu mungkin dibutuhkan nanti.

## Dokumentasi kecil yang berdampak besar
Dokumentasi tidak harus berupa buku tebal. README yang menjelaskan cara menjalankan proyek, variabel environment, script penting, struktur folder, dan alur deploy sudah sangat membantu. Komentar di kode sebaiknya menjelaskan alasan, bukan mengulang apa yang sudah jelas dari kode. Misalnya, komentar “mengurutkan array” kurang berguna jika kode sudah jelas memanggil `sort`. Komentar yang menjelaskan “urutan modul memakai angka desimal agar materi sisipan bisa ditempatkan di tengah tanpa migrasi besar” jauh lebih bermanfaat.

Untuk tim, dokumentasikan keputusan teknis penting. Mengapa memilih React Router, mengapa data disimpan di localStorage, mengapa menggunakan Cloudflare, atau mengapa validasi memakai schema tertentu. Catatan seperti ini membantu anggota baru memahami konteks dan mencegah diskusi yang sama berulang.

## Refactor secara aman
Refactor berarti mengubah struktur internal tanpa mengubah perilaku eksternal. Refactor perlu dilakukan ketika kode mulai sulit dibaca, banyak duplikasi, atau satu file terlalu besar. Namun refactor besar tanpa test berisiko tinggi. Lakukan bertahap: pahami perilaku sekarang, tambahkan test atau checklist manual, ubah bagian kecil, jalankan build, lalu commit. Jangan mencampur refactor besar dengan fitur baru jika tidak perlu, karena akan sulit meninjau perubahan dan mencari sumber bug.

Tanda kode perlu refactor antara lain: fungsi terlalu panjang, nama variabel tidak jelas, logika yang sama disalin di banyak tempat, komponen menerima terlalu banyak props, dan perubahan kecil membutuhkan edit di banyak file. Refactor bukan tentang membuat kode terlihat pintar, melainkan membuat perubahan berikutnya lebih mudah.

## Checklist kualitas sebelum rilis
Sebelum rilis, buka halaman utama di desktop dan mobile. Periksa apakah konten utama cepat terlihat, tombol dapat diklik, form punya label, gambar punya alt text, dan tidak ada error di console. Jalankan audit ringan dengan Lighthouse atau tool serupa, tetapi jangan mengejar skor tanpa memahami konteks. Skor adalah indikator, bukan tujuan akhir. Tujuan sebenarnya adalah pengalaman pengguna yang cepat, jelas, dan dapat diandalkan.

Jika Anda membangun proyek portofolio, pilar performa, aksesibilitas, dan maintainability bisa menjadi pembeda besar. Banyak website terlihat bagus pada screenshot, tetapi tidak semua nyaman digunakan dan mudah dikembangkan. Dengan memperhatikan tiga pilar ini, Anda menunjukkan cara berpikir developer yang siap bekerja pada produk nyata.
