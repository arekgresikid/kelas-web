## Mengapa design system dibutuhkan?
Saat proyek masih kecil, membuat UI sering terasa mudah. Kita menulis satu tombol, satu card, dan satu form sesuai kebutuhan halaman. Masalah muncul ketika aplikasi memiliki banyak halaman dan banyak developer. Tombol mulai punya ukuran berbeda, warna tidak konsisten, jarak antar elemen berubah-ubah, dan pola form berbeda di tiap fitur. Pengguna merasakan aplikasi kurang rapi, sementara developer menghabiskan waktu mengulang keputusan desain yang sama.

Design system adalah kumpulan prinsip, token, komponen, dan dokumentasi yang menjaga tampilan serta pengalaman tetap konsisten. Design system bukan hanya file Figma atau library komponen. Ia adalah bahasa bersama antara desain, frontend, produk, dan konten. Dengan design system, tim bisa membuat fitur baru lebih cepat karena keputusan dasar seperti warna, tipografi, spacing, radius, shadow, dan state komponen sudah tersedia.

## Design token sebagai fondasi
Design token adalah nilai desain yang diberi nama. Contohnya `color-primary`, `space-4`, `radius-lg`, `font-size-sm`, atau `shadow-card`. Token membuat desain lebih mudah diubah secara global. Jika warna brand berubah, kita mengubah token, bukan mencari ratusan class atau nilai hex di seluruh proyek. Token juga membantu menjaga konsistensi antara desain dan implementasi.

Dalam proyek dengan Tailwind, token bisa diwujudkan lewat konfigurasi tema atau variabel CSS. Dalam proyek CSS biasa, token bisa berupa CSS custom properties seperti `--color-primary` dan `--space-md`. Yang penting, token harus punya makna dan aturan penggunaan. Jangan membuat terlalu banyak token yang mirip karena akan membingungkan. Mulailah dari warna utama, warna teks, warna background, spacing dasar, radius, dan ukuran font.

## Komponen reusable yang benar-benar reusable
Komponen reusable bukan berarti komponen yang bisa melakukan semua hal. Komponen yang terlalu fleksibel sering menjadi rumit, penuh props, dan sulit dipahami. Komponen reusable yang baik punya tanggung jawab jelas. `Button` menangani tampilan dan perilaku tombol. `Input` menangani field input standar. `Card` memberi container konsisten. `Modal` mengatur dialog, fokus, dan penutupan.

Saat membuat komponen reusable, pikirkan variasi yang benar-benar dibutuhkan. Untuk Button, mungkin cukup `variant`, `size`, `disabled`, dan `isLoading`. Jangan menambahkan props untuk setiap kemungkinan style sebelum ada kebutuhan nyata. Jika sebuah halaman membutuhkan tampilan yang sangat khusus, mungkin lebih baik membuat komponen spesifik daripada memaksa komponen dasar menjadi terlalu kompleks.

## State visual komponen
Setiap komponen interaktif memiliki state visual: default, hover, active, focus, disabled, loading, error, dan success. Banyak UI terlihat bagus pada kondisi default tetapi rusak saat kondisi lain. Tombol loading harus mencegah klik ganda. Input error harus menampilkan pesan yang jelas. Focus state harus terlihat untuk pengguna keyboard. Disabled state harus tetap terbaca dan tidak hanya bergantung pada opacity yang terlalu rendah.

Mendokumentasikan state visual membantu developer menggunakan komponen dengan benar. Misalnya, Button loading sebaiknya tetap mempertahankan lebar agar layout tidak meloncat. Input error sebaiknya selalu dipasangkan dengan teks penjelasan. Modal sebaiknya menutup dengan Escape dan mengembalikan fokus ke elemen pemicu. Detail seperti ini membuat aplikasi terasa profesional.

## Konsistensi konten dan microcopy
Design system tidak hanya tentang warna dan komponen. Teks kecil di UI juga penting. Pesan tombol, label form, empty state, dan error message harus konsisten. Jika satu halaman memakai “Simpan”, halaman lain jangan tiba-tiba memakai “Submit” tanpa alasan. Jika error koneksi menggunakan gaya ramah, gunakan pola yang sama di fitur lain.

Microcopy yang baik singkat, jelas, dan berorientasi tindakan. “Data gagal disimpan. Coba lagi.” lebih baik daripada “Error.” Empty state sebaiknya menjelaskan apa yang terjadi dan apa langkah berikutnya. Label tombol sebaiknya menunjukkan aksi spesifik, seperti “Buat proyek”, “Kirim undangan”, atau “Simpan perubahan”. Konsistensi microcopy membuat aplikasi lebih mudah dipahami.

## Dokumentasi komponen
Komponen reusable perlu dokumentasi ringan. Dokumentasi dapat berupa Storybook, halaman internal, atau file markdown. Minimal jelaskan kapan komponen dipakai, props utama, contoh penggunaan, dan batasan. Dokumentasi membantu tim menghindari membuat komponen duplikat. Jika developer tahu sudah ada `Alert`, mereka tidak perlu membuat alert baru dengan style berbeda.

Dokumentasi juga mencatat keputusan. Misalnya, kapan memakai primary button versus secondary button. Primary button sebaiknya untuk aksi utama satu halaman, bukan semua tombol. Destructive button untuk aksi berisiko seperti hapus data. Link untuk navigasi, button untuk aksi. Aturan kecil seperti ini mengurangi inkonsistensi dan meningkatkan aksesibilitas.

## Komposisi lebih baik daripada konfigurasi berlebihan
Dalam React, komposisi sering lebih fleksibel daripada membuat props yang terlalu banyak. Daripada membuat `Card` dengan props `title`, `subtitle`, `image`, `footer`, `badge`, dan banyak variasi lain, kadang lebih baik membuat `Card`, `CardHeader`, `CardContent`, dan `CardFooter`. Dengan komposisi, pengguna komponen dapat menyusun isi tanpa komponen utama harus tahu semua kemungkinan.

Namun komposisi juga perlu batas. Jika terlalu bebas, konsistensi bisa hilang. Solusinya adalah menyediakan komponen dasar yang memberi struktur dan style, lalu biarkan konten diatur secara wajar. Dokumentasi contoh penggunaan akan membantu tim memahami pola yang diharapkan.

## Audit komponen yang sudah ada
Sebelum membangun design system besar, audit UI yang ada. Cari tombol yang mirip tetapi berbeda, card yang punya padding tidak konsisten, input tanpa label, warna yang tidak memakai token, dan pesan error yang tidak seragam. Kelompokkan pola yang sering muncul. Dari sana, buat komponen dasar yang menyelesaikan masalah nyata.

Jangan membuat design system sebagai proyek terpisah yang terlalu lama sebelum fitur berjalan. Mulailah kecil: Button, Input, Textarea, Select, Card, Badge, Alert, Modal, dan EmptyState. Gunakan pada fitur baru, lalu migrasikan bagian lama secara bertahap. Design system yang baik tumbuh bersama produk.

## Latihan praktis
Pilih tiga halaman di aplikasi Anda. Catat semua variasi tombol, input, card, warna, dan pesan error. Setelah itu, buat daftar komponen dasar yang bisa menyatukan pola tersebut. Tulis aturan sederhana: kapan memakai tiap variant, ukuran apa yang tersedia, dan bagaimana state error atau loading ditampilkan. Latihan ini akan melatih Anda melihat UI sebagai sistem, bukan kumpulan elemen terpisah.
