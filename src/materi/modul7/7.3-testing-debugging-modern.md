## Testing bukan hanya tugas QA
Banyak pemula menganggap testing sebagai pekerjaan tambahan yang dilakukan setelah aplikasi selesai. Dalam praktik profesional, testing adalah bagian dari cara berpikir saat membangun fitur. Tujuannya bukan membuat semua kode sempurna, melainkan memberi sinyal dini ketika perubahan baru merusak perilaku lama. Semakin besar aplikasi, semakin sulit menguji semua alur secara manual. Tombol yang terlihat sederhana bisa terhubung ke validasi form, pemanggilan API, penyimpanan lokal, routing, dan tampilan error. Tanpa test, kita mengandalkan ingatan manusia untuk mengecek semuanya setiap kali deploy.

Testing juga membantu dokumentasi. Test yang baik menjelaskan apa yang diharapkan dari sebuah fungsi atau komponen. Misalnya, jika fungsi menghitung progres belajar harus menghasilkan `100` ketika semua materi selesai, test akan menyimpan aturan itu dalam kode. Ketika orang lain mengubah implementasi, mereka bisa menjalankan test untuk memastikan aturan bisnis tetap benar.

## Jenis test yang perlu dikenal
Ada tiga kelompok test yang umum di proyek web: unit test, integration test, dan end-to-end test. Unit test memeriksa bagian kecil seperti fungsi utilitas, formatter tanggal, validasi input, atau perhitungan harga. Test ini cepat dan mudah dijalankan. Integration test memeriksa beberapa bagian yang bekerja bersama, misalnya form login yang memvalidasi input lalu memanggil fungsi autentikasi. End-to-end test memeriksa alur dari sudut pandang pengguna di browser, seperti membuka halaman, mengisi form, klik tombol, lalu melihat hasil.

Tidak semua fitur membutuhkan semua jenis test. Untuk proyek kecil, mulai dari unit test pada fungsi penting dan integration test pada komponen yang rawan rusak. End-to-end test berguna untuk alur utama seperti registrasi, login, checkout, atau pengiriman formulir kontak. Prinsipnya: test bagian yang jika rusak akan membuat pengguna gagal mencapai tujuan.

## Debugging dengan pendekatan sistematis
Debugging bukan menebak-nebak. Saat bug muncul, tulis dulu gejala yang terlihat: halaman mana, langkah apa yang dilakukan, hasil yang diharapkan, dan hasil yang terjadi. Setelah itu persempit area masalah. Apakah data dari API benar? Apakah state berubah sesuai klik pengguna? Apakah komponen menerima props yang tepat? Apakah CSS menyembunyikan elemen? Dengan memecah masalah, Anda menghindari kebiasaan mengganti kode secara acak.

Gunakan DevTools browser secara aktif. Tab Console menampilkan error JavaScript dan log. Tab Network menunjukkan request API, status code, payload, dan response. Tab Elements membantu memeriksa struktur HTML dan style CSS. Tab Application berguna untuk melihat localStorage, sessionStorage, cookie, service worker, dan cache. Untuk aplikasi React, React Developer Tools membantu melihat tree komponen, props, state, dan render yang terjadi.

## Console log yang efektif
`console.log` tetap berguna jika dipakai dengan disiplin. Jangan hanya menulis `console.log(data)` di banyak tempat tanpa label. Gunakan label yang jelas seperti `console.log('response login:', response)` atau `console.table(items)` untuk array object. Hapus log debugging sebelum commit kecuali memang dibutuhkan untuk observabilitas. Untuk error yang ditangani, gunakan pesan yang membantu, bukan sekadar `error happened`.

Saat menelusuri alur data, log titik masuk dan titik keluar. Misalnya, log input sebelum validasi, hasil validasi, request yang dikirim, response yang diterima, dan state akhir yang ditampilkan. Jika salah satu tahap berbeda dari harapan, Anda menemukan lokasi masalah. Pendekatan ini jauh lebih cepat daripada membaca seluruh file tanpa arah.

## Testing pada komponen UI
Komponen UI sebaiknya diuji berdasarkan perilaku, bukan detail implementasi. Pengguna tidak peduli nama state internal atau struktur class CSS tertentu; mereka peduli apakah tombol bisa diklik, pesan error muncul, dan data tersimpan. Karena itu, test UI yang baik mencari elemen berdasarkan teks, label, role, atau perilaku aksesibilitas. Misalnya, cari tombol dengan role `button` dan nama `Simpan`, lalu klik dan periksa apakah pesan sukses muncul.

Pendekatan ini membuat test lebih tahan terhadap refactor. Jika Anda mengganti struktur HTML tetapi perilaku pengguna tetap sama, test tidak perlu berubah. Sebaliknya, jika test terlalu bergantung pada class internal, perubahan style kecil bisa membuat test gagal padahal fitur masih berjalan.

## Error handling sebagai bagian dari kualitas
Aplikasi nyata tidak selalu berjalan di kondisi ideal. Internet bisa lambat, API bisa gagal, token bisa kedaluwarsa, input pengguna bisa tidak valid, dan data bisa kosong. Testing harus mencakup kondisi gagal, bukan hanya happy path. Tampilkan loading state saat data dimuat, pesan error saat request gagal, dan empty state saat belum ada data. Jangan biarkan pengguna menatap layar kosong tanpa penjelasan.

Buat pesan error yang membantu tindakan berikutnya. Daripada menampilkan “Terjadi kesalahan”, lebih baik “Gagal memuat progres belajar. Periksa koneksi lalu coba lagi.” Pesan seperti ini mengurangi kebingungan dan membuat aplikasi terasa lebih matang.

## Kebiasaan sebelum commit
Sebelum melakukan commit, jalankan minimal lint, build, dan test yang relevan. Lint membantu menemukan pola kode bermasalah. Build memastikan TypeScript dan bundler bisa menyelesaikan aplikasi. Test memastikan perilaku penting tidak rusak. Jika proyek belum memiliki test otomatis, lakukan checklist manual untuk alur yang Anda ubah: buka halaman, coba input valid dan tidak valid, cek tampilan mobile, dan periksa console browser.

Tuliskan pesan commit yang jelas. Pesan seperti `fix bug` kurang membantu ketika riwayat proyek dibaca kembali. Gunakan pesan yang menjelaskan perubahan, misalnya `add validation for profile form` atau `improve module progress calculation`. Kebiasaan kecil ini sangat berguna saat bekerja dalam tim atau saat harus mencari sumber masalah beberapa minggu kemudian.

## Latihan praktis
Pilih satu fitur yang sudah ada, lalu buat daftar kemungkinan gagal: data kosong, koneksi lambat, input salah, user belum login, dan server error. Setelah itu, periksa apakah UI sudah menangani semua kondisi tersebut. Jika belum, tambahkan state yang sesuai. Latihan ini melatih Anda melihat aplikasi bukan hanya dari sisi “berhasil jalan di laptop saya”, tetapi dari sisi pengguna nyata dengan kondisi yang beragam.
