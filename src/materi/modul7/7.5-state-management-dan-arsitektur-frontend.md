## Mengapa state management menjadi penting?
State adalah data yang membuat tampilan aplikasi berubah. Contohnya: user sedang login atau belum, isi keranjang belanja, tema gelap atau terang, daftar materi yang sudah selesai, nilai input form, tab yang aktif, dan hasil pencarian. Pada aplikasi kecil, state sering cukup disimpan langsung di komponen dengan `useState`. Namun ketika aplikasi berkembang, data yang sama bisa dibutuhkan oleh banyak halaman atau banyak komponen. Jika tidak dikelola dengan baik, state akan tersebar, sulit dilacak, dan rawan menghasilkan bug.

Masalah state biasanya muncul perlahan. Awalnya hanya satu komponen membutuhkan data. Lalu komponen anak juga membutuhkan data yang sama. Setelah itu halaman lain perlu membaca dan mengubah data tersebut. Kita mulai mengirim props melewati banyak lapisan komponen, membuat callback panjang, dan sulit memahami dari mana data berubah. Kondisi ini sering disebut prop drilling. Prop drilling tidak selalu buruk, tetapi jika terlalu dalam dan sering terjadi, itu tanda kita perlu memikirkan strategi state management yang lebih jelas.

## Kategori state dalam aplikasi
Sebelum memilih alat, pahami dulu jenis state. State UI adalah data yang hanya memengaruhi tampilan, seperti modal terbuka, dropdown aktif, atau tab terpilih. State form adalah data sementara dari input pengguna. State server adalah data yang berasal dari backend, seperti profil user, daftar produk, progres belajar, atau komentar. State global adalah data yang dibutuhkan banyak bagian aplikasi, seperti sesi login, preferensi tema, atau role pengguna.

Membedakan kategori ini membantu menentukan tempat penyimpanan. Tidak semua state harus masuk global store. Jika modal hanya dipakai di satu komponen, simpan di komponen tersebut. Jika data hanya dibutuhkan satu subtree, angkat state ke parent terdekat. Jika data berasal dari API dan perlu caching, pertimbangkan library khusus server state. Jika data benar-benar lintas aplikasi, gunakan Context atau store global.

## Local state, lifted state, dan Context
Strategi paling sederhana adalah local state. Gunakan ini ketika data hanya relevan untuk satu komponen. Contohnya nilai pencarian pada search box lokal atau status membuka accordion. Jika dua komponen saudara membutuhkan state yang sama, naikkan state ke parent terdekat. Ini disebut lifted state. Parent menjadi sumber kebenaran, lalu mengirim data dan fungsi update ke anak.

Context berguna untuk data yang perlu diakses banyak komponen tanpa mengirim props berulang. Contohnya tema, autentikasi, preferensi bahasa, atau konfigurasi aplikasi. Namun Context bukan solusi untuk semua masalah. Jika nilai Context sering berubah dan dipakai banyak komponen, perubahan tersebut dapat memicu render besar. Gunakan Context untuk data yang relatif stabil atau pecah menjadi beberapa Context kecil agar perubahan lebih terkontrol.

## Server state berbeda dari client state
Banyak developer pemula menyamakan semua data sebagai state biasa. Padahal data dari server memiliki masalah tambahan: loading, error, caching, refetch, stale data, pagination, optimistic update, dan sinkronisasi antar halaman. Jika aplikasi banyak bergantung pada API, library seperti TanStack Query atau SWR dapat membantu. Library semacam ini tidak hanya menyimpan data, tetapi juga mengelola status request, cache, retry, dan pembaruan data.

Contoh sederhana: halaman profil mengambil data user. Tanpa server-state library, kita harus menulis `useEffect`, `isLoading`, `error`, `setData`, pembatalan request, dan logika refetch. Untuk satu halaman mungkin masih mudah. Tetapi ketika pola itu muncul di banyak tempat, kode menjadi berulang dan rawan tidak konsisten. Dengan pendekatan server state, komponen bisa fokus pada tampilan dan perilaku, sementara mekanisme fetch dikelola oleh lapisan yang lebih cocok.

## Struktur folder yang mudah berkembang
Tidak ada satu struktur folder yang benar untuk semua proyek. Namun ada prinsip umum: letakkan file berdasarkan alasan perubahan. Untuk proyek kecil, struktur berdasarkan jenis file seperti `components`, `hooks`, `types`, dan `utils` cukup. Untuk proyek menengah, struktur berdasarkan fitur sering lebih mudah dirawat, misalnya `features/auth`, `features/course`, `features/admin`, dan `features/payment`. Di dalam setiap fitur bisa ada komponen, hook, service, dan tipe yang spesifik untuk fitur tersebut.

Struktur berdasarkan fitur membantu mengurangi efek perubahan menyebar. Jika kita mengubah alur autentikasi, sebagian besar file terkait berada di folder auth. Jika menambah fitur sertifikat, file terkait berada di folder certificate. Ini membuat onboarding lebih mudah karena developer baru bisa memahami batas fitur tanpa membuka seluruh aplikasi.

## Pola container dan presentational component
Salah satu cara menjaga komponen tetap rapi adalah memisahkan komponen yang mengurus data dari komponen yang mengurus tampilan. Komponen container mengambil data, mengelola state, dan menangani event utama. Komponen presentational menerima props dan menampilkan UI. Pola ini tidak harus diterapkan secara kaku, tetapi sangat membantu ketika komponen mulai terlalu besar.

Misalnya, `CourseProgressPage` dapat mengambil data progres, menangani refresh, dan menentukan loading state. Lalu `ProgressSummaryCard` hanya menerima `completedCount`, `totalCount`, dan `percentage`. Dengan begitu, card mudah dites, mudah dipakai ulang, dan tidak perlu tahu dari mana data berasal. Ketika sumber data berubah dari localStorage ke API, komponen presentational tidak perlu ikut berubah.

## Naming dan boundary antar modul
Nama yang baik mengurangi kebutuhan komentar. Gunakan nama yang menjelaskan maksud, bukan hanya bentuk. `calculateCompletionPercentage` lebih jelas daripada `calc`. `isAuthenticated` lebih jelas daripada `flag`. Untuk function yang melakukan efek samping, nama harus menunjukkan aksi, misalnya `saveProgress`, `fetchUserProfile`, atau `syncLocalProgress`.

Boundary antar modul juga penting. Komponen UI sebaiknya tidak langsung mengetahui detail endpoint API. Service API sebaiknya tidak mengandung logika tampilan. Hook boleh menjadi penghubung antara service dan komponen. Tipe data sebaiknya dipakai konsisten agar perubahan struktur mudah ditemukan. Ketika boundary jelas, aplikasi lebih mudah diuji dan refactor lebih aman.

## Kapan memakai global store?
Global store seperti Zustand, Redux Toolkit, Jotai, atau Recoil berguna ketika state benar-benar lintas aplikasi dan memiliki alur update yang kompleks. Contohnya editor visual, dashboard admin dengan filter global, aplikasi kolaboratif, atau state keranjang belanja yang dipakai banyak halaman. Namun memakai global store terlalu awal bisa membuat aplikasi lebih rumit dari yang diperlukan.

Sebelum memilih global store, tanyakan: apakah state ini dibutuhkan banyak area yang tidak berdekatan? Apakah update state punya aturan bisnis kompleks? Apakah Context mulai menyebabkan re-render yang sulit dikontrol? Apakah debugging perubahan state menjadi sulit? Jika jawabannya ya, store global bisa menjadi pilihan. Jika tidak, local state, lifted state, Context, dan server-state library mungkin sudah cukup.

## Latihan praktis
Ambil satu fitur di proyek Anda dan petakan semua state yang ada. Kelompokkan menjadi UI state, form state, server state, dan global state. Setelah itu, tentukan apakah masing-masing state sudah berada di tempat yang tepat. Jika ada state yang hanya dipakai satu komponen tetapi disimpan global, turunkan. Jika ada state yang dikirim melewati terlalu banyak komponen, pertimbangkan Context atau perubahan struktur. Latihan ini melatih Anda membuat keputusan arsitektur berdasarkan kebutuhan nyata, bukan sekadar mengikuti tren library terbaru.
