## API sebagai jembatan aplikasi
API adalah jembatan antara frontend dan backend. Frontend meminta data, backend memproses logika dan mengirim respons. Dalam aplikasi modern, hampir semua fitur penting menggunakan API: login, mengambil profil user, menyimpan progres, menampilkan daftar produk, memproses pembayaran, dan mengirim notifikasi. Karena itu, developer frontend perlu memahami bukan hanya cara memanggil `fetch`, tetapi juga bagaimana merancang alur data yang aman dan mudah dirawat.

Integrasi API yang buruk sering menghasilkan pengalaman pengguna yang membingungkan. Halaman kosong saat data belum datang, tombol bisa diklik berkali-kali, error tidak jelas, atau data lama tetap tampil setelah update. Integrasi yang baik selalu memikirkan empat kondisi: loading, success, empty, dan error. Keempat kondisi ini harus punya tampilan dan perilaku yang jelas.

## Memahami request dan response
Setiap request API memiliki method, URL, header, dan body. Method `GET` digunakan untuk membaca data. `POST` biasanya membuat data baru atau menjalankan aksi. `PUT` atau `PATCH` memperbarui data. `DELETE` menghapus data. Header sering membawa informasi seperti tipe konten, token autentikasi, atau konfigurasi cache. Body membawa data yang dikirim ke server, biasanya dalam format JSON.

Respons API memiliki status code dan payload. Status 200-an menandakan sukses. Status 400-an biasanya terkait kesalahan dari client, seperti input tidak valid atau user belum login. Status 500-an menunjukkan masalah di server. Frontend sebaiknya tidak memperlakukan semua error sama. Error validasi perlu ditampilkan dekat field terkait. Error autentikasi bisa mengarahkan user login ulang. Error server bisa menampilkan pesan umum dan opsi coba lagi.

## Membuat lapisan service API
Untuk proyek yang mulai berkembang, hindari menulis URL dan detail request di banyak komponen. Buat lapisan service API. Misalnya, `authService.login`, `progressService.getProgress`, atau `courseService.getModules`. Komponen cukup memanggil fungsi yang jelas, tanpa mengetahui detail endpoint. Jika endpoint berubah, perubahan dilakukan di satu tempat.

Lapisan service juga tempat yang baik untuk mengatur base URL, header default, parsing JSON, dan normalisasi error. Dengan pola ini, kode komponen menjadi lebih bersih. Komponen fokus menampilkan loading, data, dan error. Service fokus pada komunikasi dengan backend. Jika nanti Anda mengganti backend atau menambah autentikasi token, struktur proyek lebih siap.

## Loading state dan empty state
Loading state memberi tahu pengguna bahwa aplikasi sedang bekerja. Tanpa loading state, pengguna bisa mengira aplikasi rusak. Loading tidak harus selalu spinner. Untuk daftar konten, skeleton loading sering terasa lebih halus karena memberi gambaran bentuk halaman. Untuk tombol submit, ubah teks menjadi “Menyimpan...” dan nonaktifkan tombol sementara agar user tidak mengirim request berulang.

Empty state berbeda dari error. Empty state berarti request berhasil, tetapi data belum ada. Misalnya user belum memiliki proyek, belum ada komentar, atau belum menyelesaikan materi. Empty state yang baik menjelaskan situasi dan menawarkan aksi berikutnya. Contoh: “Belum ada proyek. Buat proyek pertama Anda untuk mulai membangun portofolio.” Pesan seperti ini lebih membantu daripada halaman kosong.

## Error state yang bisa ditindaklanjuti
Error state harus membantu pengguna mengambil langkah berikutnya. Jika koneksi gagal, berikan tombol coba lagi. Jika user belum login, arahkan ke halaman login. Jika input tidak valid, jelaskan field mana yang perlu diperbaiki. Jangan menampilkan detail teknis seperti stack trace kepada pengguna umum. Detail teknis cukup disimpan untuk log developer.

Penting juga untuk menghindari error yang diam-diam. Jika request gagal tetapi UI tetap terlihat sukses, pengguna bisa kehilangan kepercayaan. Misalnya tombol “Simpan” menutup modal padahal data gagal tersimpan. Lebih baik tetap tampilkan modal, beri pesan error, dan izinkan pengguna mencoba lagi.

## Optimistic update dan rollback
Optimistic update adalah teknik memperbarui UI sebelum server mengonfirmasi sukses. Contohnya ketika user klik tombol suka, angka like langsung bertambah agar aplikasi terasa cepat. Jika server kemudian gagal, UI dikembalikan atau ditampilkan pesan error. Teknik ini cocok untuk aksi yang sederhana dan sering berhasil, tetapi harus digunakan hati-hati untuk data penting seperti pembayaran atau perubahan hak akses.

Untuk menerapkan optimistic update, simpan state sebelumnya. Setelah user melakukan aksi, update tampilan. Jika request berhasil, biarkan state baru. Jika gagal, rollback ke state lama dan beri pesan. Pola ini membuat aplikasi terasa responsif tanpa mengorbankan kejelasan ketika terjadi masalah.

## Pagination, filtering, dan pencarian
Ketika data banyak, jangan memuat semuanya sekaligus. Gunakan pagination, infinite scroll, atau filter server-side. Pagination cocok untuk data yang terstruktur seperti daftar transaksi atau user admin. Infinite scroll cocok untuk feed konten, tetapi harus tetap mempertimbangkan aksesibilitas dan kemampuan pengguna menemukan posisi. Untuk pencarian, gunakan debounce agar request tidak dikirim setiap huruf tanpa jeda.

Jika filter dan pencarian memengaruhi URL query, pengguna bisa membagikan halaman dengan kondisi yang sama. Ini sangat berguna pada dashboard, katalog, atau halaman admin. Contohnya `?q=react&status=published&page=2`. Dengan menyimpan state penting di URL, aplikasi lebih mudah digunakan ulang dan navigasi browser menjadi lebih natural.

## Keamanan dasar saat bekerja dengan API
Jangan menyimpan secret backend di frontend. Semua kode frontend bisa dibaca oleh pengguna melalui browser. Token publik berbeda dari secret. Jika sebuah key memberi akses sensitif, simpan di server atau environment backend. Untuk token autentikasi user, pahami risiko localStorage, sessionStorage, dan cookie. Cookie HTTP-only sering lebih aman dari pencurian via JavaScript, tetapi tetap harus dikonfigurasi dengan benar.

Selalu validasi input di server meskipun frontend sudah melakukan validasi. Frontend dapat dimodifikasi oleh pengguna. Backend adalah lapisan terakhir yang menjaga integritas data. Untuk aplikasi yang menerima HTML atau konten bebas, waspadai XSS. Jangan menyisipkan HTML mentah tanpa sanitasi. Untuk request yang mengubah data, pahami juga risiko CSRF jika menggunakan cookie.

## Latihan praktis
Buat satu fungsi service untuk mengambil daftar data, lalu tampilkan empat kondisi UI: loading, success, empty, dan error. Tambahkan tombol coba lagi saat error. Setelah itu, buat form kecil yang mengirim data ke API palsu atau endpoint lokal. Nonaktifkan tombol saat submit berlangsung dan tampilkan pesan berhasil atau gagal. Latihan ini akan membuat Anda terbiasa membangun pengalaman data fetching yang lengkap, bukan sekadar menampilkan data ketika semuanya berjalan ideal.
