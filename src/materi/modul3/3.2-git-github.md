## Mengapa Harus Git?
Bayangkan Anda sedang menulis skripsi atau coding proyek besar. Anda sering menyimpan file dengan nama:
- `proyek-fix.zip`
- `proyek-fix-banget.zip`
- `proyek-fix-final-bismillah.zip`

Ini sangat kacau. **Git** hadir untuk menyelesaikan masalah ini. Git adalah sistem kontrol versi (*Version Control System*) yang mencatat setiap perubahan kode Anda secara rapi dan terorganisir.

---

## 1. Konsep Tiga Pilar Git
Untuk memahami Git, Anda harus paham 3 area kerja utamanya:

1.  **Working Directory**: Folder tempat Anda mengetik kode (status: *Modified*).
2.  **Staging Area**: "Ruang tunggu" sebelum perubahan disimpan. Anda memilih file mana saja yang siap (status: *Staged*).
3.  **Local Repository**: Database tempat Git menyimpan snapshot permanen kode Anda (status: *Committed*).

---

## 2. Perintah Dasar (Cheat Sheet)
Berikut adalah alur kerja harian yang akan selalu Anda gunakan:

### Persiapan Awal
- `git init`: Mengubah folder biasa menjadi "radar" Git.
- `git clone <url>`: Menyalin proyek yang sudah ada di internet ke komputer Anda.

### Alur Simpan (Save Workflow)
- `git status`: Mengecek file mana yang berubah atau belum disimpan.
- `git add <nama-file>`: Memasukkan file ke **Staging Area**.
- `git commit -m "pesan"`: Menyimpan perubahan secara permanen ke sejarah proyek.

### Sinkronisasi ke Cloud (GitHub)
- `git remote add origin <url>`: Menghubungkan komputer Anda dengan GitHub.
- `git push -u origin main`: Mengirim kode dari komputer Anda ke GitHub.
- `git pull`: Mengambil kode terbaru dari GitHub (jika ada rekan tim yang update).

---

## 3. Branching: Kekuatan Utama Git
**Branch** (Cabang) memungkinkan Anda mencoba fitur baru tanpa merusak kode utama yang sudah jalan.

- **Main Branch**: Cabang suci yang berisi kode produksi yang sudah "siap pakai".
- **Feature Branch**: Cabang tempat Anda bereksperimen. Jika gagal, Anda tinggal hapus cabangnya tanpa merusak `main`.

> **Analogi**: Bayangkan game GTA. `Main` adalah progres cerita utama Anda. `Branch` adalah saat Anda mencoba "Cheat" untuk bersenang-senang. Jika kacau, Anda tinggal *load* kembali ke progres utama.

---

## 4. GitHub: Lebih dari Sekadar Penyimpanan
Jika Git adalah "alatnya", GitHub adalah "media sosial" bagi para kode.

1.  **Repositories**: Rumah bagi proyek Anda.
2.  **Issues**: Tempat mencatat "bug" atau daftar tugas (To-Do List).
3.  **Pull Requests (PR)**: Cara elegan untuk meminta izin menggabungkan kode Anda ke proyek utama. Ini adalah tempat di mana kode Anda di-review oleh senior.
4.  **GitHub Actions**: Robot otomatis yang mengetes kode Anda secara otomatis setiap kali Anda melakukan `push`.

---

## 5. Tips Pro: Menulis Pesan Commit
Jangan menulis pesan commit seperti: `"fix"`, `"update"`, atau `"bismillah"`.
Gunakan format **Conventional Commits**:
- `feat: tambah fitur login`
- `fix: perbaiki tombol yang error di mobile`
- `docs: update panduan instalasi`

Ini membuat sejarah proyek Anda terlihat sangat profesional dan mudah dibaca oleh tim lain.

---

## 6. Menghadapi "Conflict" (Tabrakan Kode)
Conflict terjadi ketika dua orang mengedit baris kode yang sama di file yang sama. Git akan bingung: *"Punya siapa yang harus saya simpan?"*

**Jangan panik!** Ini adalah hal biasa. Cara menyelesaikannya:
1. Buka file yang bermasalah.
2. Anda akan melihat tanda `<<<<<<< HEAD`.
3. Pilih kode mana yang mau dipertahankan (milik Anda, milik rekan, atau keduanya).
4. Hapus tanda-tanda aneh dari Git, simpan file, lalu lakukan `commit` ulang.
