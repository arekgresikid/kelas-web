## Mengapa Harus Git?
Bayangkan Anda sedang menulis skripsi atau coding proyek besar. Anda sering menyimpan file dengan nama:
- `proyek-fix.zip`
- `proyek-fix-banget.zip`
- `proyek-fix-final-bismillah.zip`

Ini sangat kacau. **Git** hadir untuk menyelesaikan masalah ini. Git adalah sistem kontrol versi (*Version Control System*) yang mencatat setiap perubahan kode Anda secara rapi dan terorganisir.

---

## 1. Konsep Tiga Pilar Git
Untuk memahami Git, Anda harus paham 3 area kerja utamanya:

1.  **Working Directory**: Folder tempat Anda mengetik kode (status: *Modified*).
2.  **Staging Area**: "Ruang tunggu" sebelum perubahan disimpan. Anda memilih file mana saja yang siap (status: *Staged*).
3.  **Local Repository**: Database tempat Git menyimpan snapshot permanen kode Anda (status: *Committed*).

---

## 2. Perintah Dasar (Cheat Sheet Lengkap)
Gunakan daftar ini sebagai contekan saat Anda lupa perintah terminal:

### ⚡ Persiapan & Konfigurasi
- `git init`: Membuat folder proyek baru diawasi oleh Git.
- `git clone <url>`: Mendownload proyek dari GitHub ke komputer Anda.
- `git config --global user.name "Nama Anda"`: Setting identitas pengirim.
- `git config --global user.email "email@anda.com"`: Setting email pengirim.
- `git remote -v`: Mengecek daftar alamat repo online yang terhubung.

### 💾 Alur Kerja Harian (The Loop)
- `git status`: Melihat file mana yang baru, diedit, atau siap simpan.
- `git add .`: Menandai **semua** file yang berubah untuk disimpan.
- `git add <nama-file>`: Hanya menandai satu file spesifik.
- `git commit -m "pesan"`: Menyimpan snapshot secara permanen.
- `git log --oneline --graph`: Melihat sejarah proyek dengan visual cabang yang rapi.
- `git diff`: Melihat detail baris mana yang Anda ubah sebelum di-add.
- `git diff --staged`: Melihat perubahan yang sudah di-add tapi belum di-commit.

### 🌿 Branching & Merging
- `git branch`: Melihat daftar cabang.
- `git branch -d <nama-cabang>`: Menghapus cabang yang sudah tidak terpakai.
- `git checkout -b <nama-cabang>`: Membuat cabang baru dan langsung pindah ke sana.
- `git checkout <nama-cabang>`: Pindah ke cabang yang sudah ada.
- `git merge <nama-cabang>`: Menggabungkan perubahan ke cabang saat ini.
- `git switch -`: Jalan pintas untuk pindah ke cabang sebelumnya.

### 🚀 Sinkronisasi Cloud (GitHub)
- `git remote add origin <url>`: Menghubungkan folder lokal dengan repo di GitHub.
- `git push -u origin main`: Mengirim kode pertama kali (sekaligus setting default).
- `git push`: Mengirim update kode selanjutnya.
- `git pull`: Mengambil dan menggabungkan update terbaru dari rekan tim.
- `git fetch`: Mengambil info update dari GitHub tanpa menggabungkan (aman untuk cek dulu).

### 🛠️ Perintah Penyelamat (Undo/Fix)
- `git checkout -- <file>`: Membatalkan editan (kembali ke kondisi terakhir disimpan).
- `git reset HEAD <file>`: Membatalkan `git add` (keluar dari ruang tunggu).
- `git reset --hard HEAD~1`: Menghapus commit terakhir (Hati-hati! Data akan hilang).
- `git commit --amend`: Memperbaiki pesan commit terakhir atau menambah file yang lupa di-add.
- `git stash`: Menyembunyikan sementara kerjaan yang belum selesai.
- `git stash pop`: Mengambil kembali kerjaan yang disembunyikan tadi.

### 🔍 Inspeksi & Pembersihan
- `git blame <file>`: Melihat siapa yang menulis baris kode tertentu (untuk investigasi bug).
- `git show <commit-id>`: Melihat detail perubahan pada commit tertentu.
- `git clean -fd`: Menghapus file-file sampah yang tidak diawasi oleh Git.
- `git tag -a v1.0 -m "Versi 1.0"`: Memberikan label versi pada titik tertentu di sejarah.

## 3. Branching: Kekuatan Utama Git
**Branch** (Cabang) memungkinkan Anda mencoba fitur baru tanpa merusak kode utama yang sudah jalan.

- **Main Branch**: Cabang suci yang berisi kode produksi yang sudah "siap pakai".
- **Feature Branch**: Cabang tempat Anda bereksperimen. Jika gagal, Anda tinggal hapus cabangnya tanpa merusak `main`.

> **Analogi**: Bayangkan game GTA. `Main` adalah progres cerita utama Anda. `Branch` adalah saat Anda mencoba "Cheat" untuk bersenang-senang. Jika kacau, Anda tinggal *load* kembali ke progres utama.

---

## 4. GitHub: Lebih dari Sekadar Penyimpanan
Jika Git adalah "alatnya", GitHub adalah "media sosial" bagi para kode.

1.  **Repositories**: Rumah bagi proyek Anda.
2.  **Issues**: Tempat mencatat "bug" atau daftar tugas (To-Do List).
3.  **Pull Requests (PR)**: Cara elegan untuk meminta izin menggabungkan kode Anda ke proyek utama. Ini adalah tempat di mana kode Anda di-review oleh senior.
4.  **GitHub Actions**: Robot otomatis yang mengetes kode Anda secara otomatis setiap kali Anda melakukan `push`.

---

## 5. Tips Pro: Menulis Pesan Commit
Jangan menulis pesan commit seperti: `"fix"`, `"update"`, atau `"bismillah"`.
Gunakan format **Conventional Commits**:
- `feat: tambah fitur login`
- `fix: perbaiki tombol yang error di mobile`
- `docs: update panduan instalasi`

Ini membuat sejarah proyek Anda terlihat sangat profesional dan mudah dibaca oleh tim lain.

---

## 6. Menghadapi "Conflict" (Tabrakan Kode)
Conflict terjadi ketika dua orang mengedit baris kode yang sama di file yang sama. Git akan bingung: *"Punya siapa yang harus saya simpan?"*

**Jangan panik!** Ini adalah hal biasa. Cara menyelesaikannya:
1. Buka file yang bermasalah.
2. Anda akan melihat tanda `<<<<<<< HEAD`.
3. Pilih kode mana yang mau dipertahankan (milik Anda, milik rekan, atau keduanya).
4. Hapus tanda-tanda aneh dari Git, simpan file, lalu lakukan `commit` ulang.
