Amazon menemukan bahwa setiap penundaan (*delay*) 100 milidetik dalam pemuatan halaman web mereka mengakibatkan penurunan penjualan sebesar 1%. Google menemukan bahwa penundaan setengah detik menyebabkan penurunan *traffic* pencarian sebesar 20%. Di tingkat *Enterprise*, Kecepatan (Performance) bukan lagi sekadar fitur kenyamanan; ia adalah **Uang**.

Selain memengaruhi tingkat pentalan (*Bounce Rate*) pengguna, performa web kini adalah salah satu faktor metrik utama yang digunakan oleh Google Search untuk merangking website Anda di mesin pencari (SEO).

Dalam materi pamungkas ini, kita akan membahas cara mendiagnosis dan mengoptimalkan performa web Anda (WPO - *Web Performance Optimization*).

## 1. Alat Ukur: Core Web Vitals (CWV)

Anda tidak bisa mengoptimalkan apa yang tidak bisa Anda ukur. Google telah menetapkan standar emas pengujian performa yang dinamakan **Core Web Vitals**. Anda dapat mengukur website Anda melalui tool gratis Google: **Lighthouse** (ada di *Chrome Developer Tools* pada tab *Lighthouse*) atau di situs *PageSpeed Insights*.

Core Web Vitals berfokus pada 3 pilar UX (User Experience):

### A. LCP (Largest Contentful Paint) - Mengukur Waktu Muat (Loading)
LCP mengukur seberapa cepat **elemen visual terbesar** (biasanya gambar sampul/Hero Image atau blok teks H1) dirender dengan sempurna di layar pengguna.
- **Standar Bagus:** Di bawah 2.5 detik.
- **Masalah Umum:** Gambar Hero terlalu besar ukurannya (resolusi 4K untuk layar HP), atau teks terblokir karena mengunduh font web kustom yang lambat.

### B. INP (Interaction to Next Paint) - Mengukur Interaktivitas / Responsivitas
*(INP menggantikan metrik lawas bernama FID pada tahun 2024)*.
INP mengukur seberapa lama jeda (*lag*) ketika pengguna melakukan interaksi (misal: Mengklik tombol "Beli", atau mengetik di *keyboard*), hingga halaman secara visual merespons (misal: tombol berubah warna atau teks muncul).
- **Standar Bagus:** Di bawah 200 milidetik (0.2 detik).
- **Masalah Umum:** Anda menjalankan *Script* JavaScript yang sangat berat di *Main Thread* browser, sehingga browser membeku (Freeze) dan mengabaikan klik pengguna.

### C. CLS (Cumulative Layout Shift) - Mengukur Stabilitas Visual
CLS mengukur loncatan visual yang tak terduga. Pernahkah Anda hendak mengklik sebuah link, tapi tiba-tiba sebuah iklan raksasa *loading* dan mendorong link tersebut ke bawah, sehingga Anda tidak sengaja mengklik iklannya? Itulah CLS yang buruk.
- **Standar Bagus:** Skor di bawah 0.1.
- **Masalah Umum:** Menampilkan gambar tanpa mendefinisikan tinggi (Height) dan lebarnya (Width) di HTML, sehingga browser tidak tahu berapa banyak ruang kosong yang harus disediakan sebelum gambar selesai diunduh.

## 2. Strategi 1: Code Splitting & Lazy Loading

Aplikasi modern seperti React sering menghasilkan satu bundel JavaScript raksasa (misal: `bundle.js` sebesar 2MB). Padahal pengguna yang mampir ke halaman "Home" sama sekali tidak butuh JavaScript untuk merender halaman "Admin Dashboard" yang kompleks.

### Code Splitting (Pemisahan Kode)
Ini adalah praktik memecah bundel 2MB tersebut menjadi potongan-potongan kecil (misal 50 file berukuran 40KB). Jika pengguna masuk ke halaman "Home", browser HANYA akan mengunduh JavaScript yang relevan untuk halaman "Home" saja.

Di Next.js (App Router), Code Splitting terjadi **secara otomatis per-rute**. Itulah mengapa Next.js secara *default* sudah sangat cepat.

### Lazy Loading Komponen
Bagaimana jika di halaman "Home" Anda ada sebuah Chart Interaktif yang sangat berat (menggunakan pustaka D3.js atau Recharts), tetapi letaknya ada di paling bawah halaman (pengguna harus men-scroll sangat jauh ke bawah untuk melihatnya)?

Jangan paksa browser mengunduh pustaka Chart tersebut di awal. Gunakan *Lazy Loading*.
Di Next.js (dan React `Suspense`), Anda bisa mengimpor komponen secara dinamis:

```tsx language-typescript
import dynamic from 'next/dynamic';

// Komponen berat ini TIDAK akan didownload saat halaman pertama kali muat
// Ia baru akan diunduh secara asinkron saat user men-scroll mendekatinya
const HeavyChart = dynamic(() => import('@/components/HeavyChart'), {
  loading: () => <p>Memuat grafik kompleks...</p>, // Placeholder
  ssr: false // Jika pustaka ini tidak mendukung render di server
});

export default function Home() {
  return (
    <div>
      <h1>Artikel Ringan</h1>
      {/* Banyak paragraf teks di sini */}
      
      {/* Di paling bawah layar */}
      <HeavyChart />
    </div>
  );
}
```

## 3. Strategi 2: Optimasi Aset (Gambar & Font)

Lebih dari 60% berat rata-rata halaman web di internet berasal dari file Gambar. Jika Anda bisa menguasai teknik optimasi aset, skor LCP Anda akan meroket ke angka 100.

### Optimasi Gambar Mutlak
Jangan pernah mengirim file `.jpg` berukuran 3MB apalagi gambar `.png` tanpa kompresi kepada klien. 
1. **Format Modern:** Selalu konversi gambar ke format **WebP** atau **AVIF**. Kualitas visualnya nyaris identik dengan JPEG namun ukuran filenya bisa 70% lebih kecil.
2. **Next/Image:** Di Next.js, JANGAN PERNAH gunakan tag HTML bawaan `<img />`. Selalu gunakan komponen bawaan Next.js `<Image />`.
   - Next.js akan secara otomatis melakukan kompresi ke format WebP di server (*On-the-fly*).
   - Ia akan melakukan *Lazy Loading* (gambar di bawah lipatan layar tidak akan diunduh sampai di-*scroll*).
   - Mencegah CLS secara otomatis karena memaksa Anda memasukkan prop `width` dan `height`.

```tsx language-typescript
import Image from 'next/image';

export default function Profil() {
  return (
    // Gambar ini akan dikompres, dilazy-load, dan diubah ke webp otomatis
    <Image 
      src="/hero-banner.jpg" 
      alt="Banner Utama" 
      width={1200} 
      height={600} 
      priority={true} // Jika ini adalah elemen LCP (Gambar terbesar di atas), set priority ke true agar tidak di-lazy-load
    />
  );
}
```

### Optimasi Font
Mengunduh *Font* kustom (seperti Google Fonts) adalah salah satu penyebab utama *Render-Blocking* (Teks tidak mau muncul sebelum Font terunduh sepenuhnya). 
- Di Next.js, gunakan modul `next/font/google`. Modul ini akan mengunduh font Google langsung di tahap kompilasi (*Build Time*) dan menyimpannya bersama aplikasi Anda (Self-Hosted), sehingga tidak ada tambahan koneksi HTTP ke server eksternal milik Google saat web dimuat.

## 4. Strategi 3: Caching Ekstrem di Ujung Jaringan (CDN/Edge)

Cara paling cepat untuk mengambil data dari server adalah... dengan **TIDAK MENGAMBILNYA** dari server sama sekali.

**Caching** adalah seni menyimpan salinan data yang sudah dihitung, sehingga permintaan berikutnya bisa dilayani secara instan tanpa proses komputasi yang berat.

### The CDN (Content Delivery Network)
Sebuah server Anda yang berada di New York (Amerika Serikat) akan memiliki ping sebesar 250ms jika diakses oleh pengguna dari Indonesia. Jarak fisik cahaya dalam serat optik tidak bisa dibohongi.

Sebuah CDN (seperti Cloudflare) memecahkan masalah fisika ini. CDN memiliki ribuan komputer (disebut *Edge Nodes*) yang tersebar di ratusan kota di seluruh dunia, termasuk Jakarta.

1. Pengguna pertama dari Jakarta mengakses web Anda. Komputer CDN Cloudflare Jakarta menelpon Server New York Anda untuk mengambil HTML. (Proses lambat).
2. CDN Jakarta lalu menyalin (Cache) HTML tersebut ke dalam memorinya.
3. Saat 1 Juta pengguna lain dari Indonesia mencoba mengakses web Anda di detik berikutnya, mereka tidak akan pernah menelpon New York. CDN Jakarta yang akan langsung melemparkan salinan HTML statis tersebut ke pengguna. Proses ini memakan waktu kurang dari 10 milidetik!

Inilah kekuatan sejati di balik arsitektur situs statis (SSG - *Static Site Generation*) di Next.js. Halaman web yang telah dirender statis di server akan ditendang ke seluruh ujung dunia melalui infrastruktur Vercel/Cloudflare. Ini membuatnya seakan-akan kebal dari DDoS dan super cepat.

## Kesimpulan: Performa adalah Pola Pikir

WPO bukan sekadar tentang *tools* atau metrik; melainkan tentang empati kepada pengguna (*Empathy for the User*). Tidak semua orang memiliki koneksi 5G atau memegang iPhone versi terbaru. Ada jutaan pengguna di area pelosok dengan sinyal 3G yang tidak stabil dan memakai ponsel pintar murah dengan RAM kecil.

Seorang *Senior Engineer* memahami bahwa setiap kilobyte dari JavaScript yang dikirimkannya membawa beban berat bagi perangkat pengguna. Dengan menggabungkan Server Components, otomatisasi gambar, Lazy Loading, dan CDN, Anda memastikan bahwa hasil karya Anda (Aplikasi Web) inklusif dan dapat diakses dengan secepat kilat oleh siapa pun, di belahan bumi mana pun.
